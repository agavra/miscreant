# 0001 - Miscreant

- **Status:** Draft
- **Author(s):** Almog Gavra

## Summary

This document covers the big picture design for Miscreant, a git-fs backed
by object storage. Miscreant can be deployed as a stateless server and
supports the "smart" git protocol over HTTP(S).

## Motivation

Agents make exceptional use of file systems, and it is becoming increasingly
clear that having access to a remote file system that is versioned and can
handle conflicts with multiple agents concurrently accessing the same files
is valuable infrastructure.

Miscreant is a low-level implementation of git that allows for concurrent
access to a git-backed file system backed by object storage. Unlike using
an existing Git hosting solution like GitHub, Miscreant offers no rate limits
and is designed to be extremely cheap to store enormous amounts of data.

Miscreant is not a complete developer product, instead the intention is to
serve as the backbone for products like [Mesa](https://mesa.dev) and 
[Fresstyle Git](https://www.freestyle.sh/docs/git).

## Goals

- implement the git HTTPs server protocol in a single deployable rust binary
- backed by object storage for cheap storage and a dissaggregated architecture
- support efficient retreival and storage of files

## Non-Goals

- replace the git client (clients will typically intract with Miscreant 
  via the `git` CLI)
- cover all the details of git protocols that are publicly documented
  elsewhere (reference are provided inline and in the final section)

## Design

### Overview

Miscreant is a single binary rust server built on [SlateDB](https://slatedb.io)
that speaks the git HTTPs protocol.

```ascii-art
┌───────────┐         ┌───────────┐       ┌S3───────────┐
│   agent   │────┐    │ Miscreant │█      │┌───────────┐│█
└───────────┘    │    │  Server   │█      ││  blobs/   ││█
┌───────────┐    │    │           │█      │└───────────┘│█
│   agent   │────┼───▶│┌─────────┐│──────▶│┌───────────┐│█
└───────────┘    │    ││ slatedb ││█      ││ slatedb/  ││█
┌───────────┐    │    │└─────────┘│█      │└───────────┘│█
│   human   │────┘    └───────────┘█      └─────────────┘█
└───────────┘          █████████████       ███████████████
```

Agents and humans can both concurrently connect to the Miscreant and use it to
coordinate and resolve conflicts. The data is stored in two locations: all
git objects (trees, commits, blob IDs) are stored in SlateDB as well as any
blob content that is under a certain threshold (64KB by default; see
[Appendix A](#appendix-a-inline-threshold-sizing) for how this default was
chosen) while all blobs that exceed this threshold are stored directly in S3. The blobs and
the slatedb directories are stored within the same S3 bucket.

Note that S3 is a placeholder for any object storage system, we could use
GCS, R2, Tigris, etc...

### Storage

SlateDB will be the main content addressed storage backend. Keys are
scoped to a repository and namespaced by segment, so a single store can
host many repositories side by side. We use four SlateDB
[segments](https://slatedb.io/rfcs/0024-segment-oriented-compaction/): a
metadata table, an object store, a ref store, and a commit-graph table.

#### Key Preamble

Every key in every segment begins with the same fixed-width preamble:

```ascii-art
┌─────────┬─────────┬─────────┬────────────────────────────────┐
│ segment │ version │ repo_id │ segment-specific key           │
│ 1 byte  │ 1 byte  │ 8 bytes │ variable                       │
└─────────┴─────────┴─────────┴────────────────────────────────┘
```

| field | type | description |
|-------|------|-------------|
| `segment` | `u8` | the segment prefix: `0x00` metadata, `0x01` object, `0x02` ref, `0x03` commit-graph |
| `version` | `u8` | the key format version |
| `repo_id` | `u64` | opaque, fixed-width repository id assigned at repo creation. |

The segment byte leads the key because segments are defined by key
prefix: a single leading byte yields exactly four segments (LSM trees)
shared by all repositories, whereas a segment byte following `repo_id`
would carve out one prefix — and thus one segment — per
`(repo, segment)` pair. The version byte versions the remainder of the
key within its segment.

`repo_id = 0x00` is reserved for global metadata (see below), which
must be readable before any specific repo is resolved. Because the SHA width (20
vs 32 bytes) is baked into object, ref, and commit-graph keys, a request must
resolve configuration before it can read those segments. The metadata segment
has fixed string keys and its values are always readable knowing nothing about
the repository yet. 

Per request:

1. **name → repo_id** — look up the repository by name in server-global
   metadata (`repo_id = 0`, key `repo:<name>`).
2. **repo_id → object-format** — read that repo's `object-format` from
   its metadata segment to learn the SHA width.
3. **read/write data segments** — object, ref, and commit-graph keys are
   now parseable at the known width.

#### Metadata Storage

**Key Layout**

```ascii-art
┌─────────┬─────────┬─────────┬────────────────────────────────┐
│ segment │ version │ repo_id │ meta_key                       │
│ 1 byte  │ 1 byte  │ 8 bytes │ variable (UTF-8)               │
└─────────┴─────────┴─────────┴────────────────────────────────┘
```

| field | type | description |
|-------|------|-------------|
| `segment` | `u8` | the segment prefix (`0x00` metadata table) |
| `version` | `u8` | the key format version |
| `repo_id` | `u64` | repository id, or `0` for server-global metadata |
| `meta_key` | `[u8]` | UTF-8 identifier of the setting. Never a SHA — this segment must be readable without knowing the object format. |

**Value Layout**

```ascii-art
┌─────────────┬────────────────────────────┐
│ value_type  │ value                      │
│   1 byte    │ variable                   │
└─────────────┴────────────────────────────┘
```

| field | type | description |
|-------|------|-------------|
| `value_type` | `u8` | how to interpret `value`: `0x01` UTF-8 string, `0x02` u8, `0x03` u32, `0x04` raw bytes |
| `value` | `[u8]` | the setting's value, interpreted per `value_type` |

Representative keys:

| `meta_key` | scope | `value_type` | description |
|-----------|-------|--------------|-------------|
| `repo:<name>` | global (`repo_id = 0`) | `0x03` u32 / raw | maps a repository name to its `repo_id` |
| `object-format` | per-repo | `0x01` string | `sha1` or `sha256`; sets the SHA width for this repo's data segments |
| `default-branch` | per-repo | `0x01` string | e.g. `refs/heads/main`; the symref target advertised for `HEAD` |
| `schema-version` | per-repo | `0x03` u32 | version of the overall KV layout, for future migrations |

#### Git Object Storage

**Key Layout**

```ascii-art
┌─────────┬─────────┬─────────┬────────────────────────────────┐
│ segment │ version │ repo_id │ object_id                      │
│ 1 byte  │ 1 byte  │ 8 bytes │ 20 bytes (SHA-1) / 32 (SHA-256)│
└─────────┴─────────┴─────────┴────────────────────────────────┘
```

| field | type | description |
|-------|------|-------------|
| `segment` | `u8` | the segment prefix (`0x01` object table) |
| `version` | `u8` | the key format version |
| `repo_id` | `u64` | repository id |
| `object_id` | `20|32 bytes` | the binary object SHA |

**Value Layout**

```ascii-art
┌─────────────┬────────────────────────────┐
│ object_tag  │ payload                    │
│   8 bits    │ variable                   │
└─────────────┴────────────────────────────┘
```

| field | type | description |
|-------|------|-------------|
| `object_tag` | `u8` | `0x01` BlobInline, `0x02` BlobPointer, `0x03` Tree, `0x04` Commit, `0x05` Tag |
| `payload` | `[u8]` | Serialized payload |

Inline blob payloads store the object's canonical git encoding `<type>
<size>\0<content>`. A pointer's content lives in object storage under
`blobs/<xx>/<rest>`, where `<xx>` is the first two hex characters of the object
SHA and `<rest>` is the remainder (mirroring git's loose-object fanout), so the
location is derivable from the key. The pointer's payload is the blob size as
a `u64`, so `blob:limit` filters and `object-info` size queries never need an
S3 round trip.

#### Ref Storage

**Key Layout**

```ascii-art
┌─────────┬─────────┬─────────┬────────────────────────────────┐
│ segment │ version │ repo_id │ ref_name                       │
│ 1 byte  │ 1 byte  │ 8 bytes │ variable (UTF-8)               │
└─────────┴─────────┴─────────┴────────────────────────────────┘
```

| field | type | description |
|-------|------|-------------|
| `segment` | `u8` | the segment prefix (`0x02` ref table) |
| `version` | `u8` | the key format version |
| `repo_id` | `u64` | repository id |
| `ref_name` | `[u8]` | the full ref name as UTF-8, slashes included (e.g. `refs/tags/v0.9.2`, `refs/heads/main`, `HEAD`) |

**Value Layout**

```ascii-art
┌─────────────┬────────────────────────────┐
│  ref_tag    │ target                     │
│   8 bits    │ variable                   │
└─────────────┴────────────────────────────┘
```

| field | type | description |
|-------|------|-------------|
| `ref_tag` | `u8` | Ref Type `0x01` Direct, `0x02` Reference |
| `target` | `[u8]` | Interpreted per ref type: a binary object SHA when Direct, or a UTF-8 ref name when Reference |

#### Commit Graph Storage

**Key Layout**

```ascii-art
┌─────────┬─────────┬─────────┬────────────────────────────────┐
│ segment │ version │ repo_id │ commit_id                      │
│ 1 byte  │ 1 byte  │ 8 bytes │ 20 bytes (SHA-1) / 32 (SHA-256)│
└─────────┴─────────┴─────────┴────────────────────────────────┘
```

| field | type | description |
|-------|------|-------------|
| `segment` | `u8` | the segment prefix (`0x03` commit-graph table) |
| `version` | `u8` | the key format version |
| `repo_id` | `u64` | repository id |
| `commit_id` | `20|32 bytes` | the binary SHA of the commit this record describes |

**Value Layout**

```ascii-art
┌────────────┬────────────┬───────────────┬────────────────────────┐
│ generation │ root_tree  │  parent_count │ parents                │
│  4 bytes   │ 20|32 bytes│    1 byte     │ parent_count × (20|32) │
└────────────┴────────────┴───────────────┴────────────────────────┘
```

| field | type | description |
|-------|------|-------------|
| `generation` | `u32` | the generation number: `1 + max(parent generations)`, or `1` for a root commit |
| `root_tree` | `20|32 bytes` | binary SHA of the commit's root tree (Phase 2 entry point) |
| `parent_count` | `u8` | number of parents: `0` for a root, `1` for normal, `2+` for merges |
| `parents` | `[20|32 bytes]` | `parent_count` binary parent SHAs, in the commit's parent order |

> **Note:** The commit-graph segment is *derived, rebuildable* data — it
> is not content-addressed (the key is a commit SHA, but the value is
> graph metadata computed from the commit objects, not the hashed
> content). It can be regenerated wholesale from segment `0x01` at any
> time, and must be extended as new commits arrive on push.

### Transfer Protocol

The server will be implemented as a `tokio-rs/axum` server. The transfer protocol
will only support HTTP(S) for the initial implementation, and the repo will be 
specified in the path (e.g. `GET /<repo-path>/info/refs?service=git-upload-pack`)

#### Packfiles

Since SlateDB already packs keys into blocks / SSTs and compresses efficiently
(enough for object storage anyway), v1 will not store packfiles. Since packfiles
are part of the Git protocol, however, we still need to handle them.

The approach we'll take for v0 of Miscreant is to unpack the packfiles and explode
them on the receiving end. On the send side we'll pack objects after traversing the
tree set when we send them over the protocol.

#### Fetch API

We will implement only the v2 git API for the first revision of Miscreant. We will
only support requests with `Git-Protocol: version=2`.

See the [v2 protocol specification](https://git-scm.com/docs/protocol-v2) for
specific definitions of the commands. This document will only cover interesting
implementations bits as they relate to the storage format.

The list of capbilities that the server implements will be a hard coded
configuration of the server. The initial MVP capabilities are:

- `version 2`
- `agent`
- `ls-refs`
- `fetch` (including the `filter` argument, i.e. partial clone)
- `object-format`

##### `command=ls-refs`

If no optional parameters are included, this will scan he entire reftable
(segment `0x02`) and return all known references. If `symref` is specified,
then all symbolic references (via the `0x02` reference byte indicator) will
be included in the response (the query execution remains the same). The same
recursive behavior can be triggered when requesting `peel`.

Specifying a `ref-prefix <prefix>` will use a prefix scan on the reftable using
the specified `<prefix>`.

##### `command=fetch`

This implements the commit walk algorithm, which has three phases: the phase
that discovers the required commits, then the collect stage that finds the
SHAs that need to be packed, and finally the contents of the blobs in
a fetch stage.

**Phase I: Commit Discovery**

```rust
fn select_commits(self, wants, haves) {
    frontier = MaxHeap(key = generation)
    state = {} // SHA -> WANT | HAVE 
    want = 0 // how many commits we still need to serve

    for h in haves: want += mark(frontier, state, h, HAVE)
    for w in wants: want += mark(frontier, state, w, WANT)

    result = []
    done = {}

    while !frontier.is_empty() && want > 0 {
        sha = frontier.pop()
        if done.contains(sha): continue
        done.add(sha)

        if state[sha] == WANT:
            result.append(sha)
            want -=1

        commit = self.index.get_commit(sha)
        for parent in commit.parents:
            want += mark(frontier, state, parent, state[sha])
    }
}

// adds this commit to the seen state and indicates
// whether or not we need to continue traversing the
// parents of this SHA
fn mark(self, frontier, state, sha, flag) -> i32 {
    let mut want = 0;

    // state[sha] is the current state for this SHA
    // while the flag is the status of the child of this 
    // commit
    let resolved = match (state[sha], flag]) {
        (Some(HAVE), _) => return 0,
        
        // if we have the child, then by definition we
        // have the entire ancestry of the child, so we
        // can mark any ancestor as HAVE
        (Some(WANT), HAVE) => {
            want -=1;
            HAVE
        },
        (Some(WANT), WANT) => return 0;

        (None, HAVE) => HAVE
        (None, WANT) => {
            want +=1;
            WANT
        }
    }

    state[sha] = resolved;
    self.frontier.push(self.index.get_commit(sha));
    want
}
```

This uses the commit index to efficiently discover parents
and generations for every commit. If the git index is not
available, we first rebuild it by fully scanning the tree
strating from the parent reference.

**Phase II: Collect Objects**

```rust
fn collect(self, to_send, haves):
    let mut objects = OrderedSet::new();

    // Expand the have-frontier: the client has EVERYTHING reachable
    // from its have-commits, so collect every tree and blob SHA under
    // each have root — not just the roots. (The root tree of a new
    // commit almost always differs from its parent's root tree, so
    // pruning on root trees alone would essentially never fire and an
    // incremental fetch would degenerate into a full snapshot.)
    let mut have_objects = {};
    for h in haves:
        self.expand_haves(self.index.get_commit(h).root_tree, &mut have_objects)

    for commit_sha in to_send {
        objects.add(commit_sha)
        let root = self.index.get_commit(commit_sha).root_tree
        self.walk_tree(root, &mut objects, &have_objects)
    }

    objects

// Recursively insert every tree + blob SHA reachable from `tree_sha`
// into `have_objects`. The set doubles as the memoization/seen set: a
// subtree shared between several haves is visited only once, so the
// total cost is O(distinct objects reachable from the haves), not
// O(haves × snapshot).
fn expand_haves(self, tree_sha, have_objects) {
    if have_objects.contains(tree_sha): return
    have_objects.add(tree_sha)

    for (mode, name, child_sha) in self.read_tree(tree_sha) {
        match is_directory(mode) {
            true  => self.expand_haves(child_sha, have_objects),
            false => have_objects.add(child_sha)
        }
    }
}

// Recursively collect the trees + blobs under `tree_sha`, skipping
// anything the client already has. Content-addressing guarantees an
// identical tree SHA means identical content all the way down, so a
// shared tree can be pruned WITHOUT descending into it.
fn walk_tree(self, tree_sha, objects, have_objects) {
    match (objects.contains(tree_sha), have_objects.contains(tree_sha)) {
        (true, _) => return,
        // Client has this exact tree ⇒ has every tree + blob beneath it.
        // Prune the whole subtree in one SHA comparison.
        (_, true) => return,
        (false, false) => {}
    }
    objects.add(tree_sha)

    for (mode, name, child_sha) in self.read_tree(tree_sha) {
        match is_directory(mode) {
            true  => self.walk_tree(child_sha, objects, have_objects),
            false => {
                if !have_objects.contains(child_sha):   // a blob the client lacks
                    objects.add(child_sha)
            }
        }
    }
}
```

This exclusively uses point lookups on the underlying slatedb instance when
looking up tree SHAs.

Expanding the have-frontier up front costs a full tree walk over the haves'
snapshots, which for a typical incremental fetch is the same order as a
single snapshot walk. In exchange, the want-side walk prunes at the first
tree *or blob* the client already has, so the cost and output of an
incremental fetch is proportional to the size of the diff plus the size of
the have frontier — not the size of the snapshot. The `objects` set plays
the same memoization role on the want side, deduplicating trees shared
among the commits being sent.

> **Note:** A natural optimization is to cache the expanded have-set keyed
> by commit SHA — haves are almost always recent branch tips, so this cache
> is small and extremely hot. v1 recomputes the expansion per request and
> defers the cache.

**Phase III: Fetch Contents**

Now that we have all the tree SHAs, we can fetch the raw blobs that we need
to send over. This simply collects all the blob IDs and fetches them, either
inline from SlateDB for small blobs or from object storage for larger ones.

**Filters (partial clone)**

A `filter` argument is applied as a predicate inside the Phase II walk:
`blob:none` collects trees but never blobs (and `expand_haves` skips blobs
too), while `blob:limit=<n>` reads the size from the inline header or the
pointer payload. 

So that partial clones can backfill, `fetch` accepts wants for arbitrary
OIDs (git's `allowAnySHA1InWant` behavior), not just ref tips. When the
wants are blobs or trees, Phases I and II are skipped entirely and the
response is a point lookup per want, packed directly. 

##### `command=object-format`

This command will read from the metadata for the repository and return the
object format (e.g. `SHA-1`). 

##### `comand=object-info`

This command just does a point lookup for the supplied `oid` and returns the
information in the standard v2 response format.

### Receive API

The receive API [specification](https://git-scm.com/docs/http-protocol) for
the HTTP protocol is quite a bit simpler than the fetch API. Similarly to
the fetch API, there is a discovery endpoint which will return the capabilities
of the server. Again these are statically configured per repository and stored
in the repo metadata segment.

The `POST /<repo>/git-receive-pack` endpoint is what has the PUSH side of the
git protocol. It will start by unpacking the packfile and store all the objects
into a local temporary directory. This allows us to validate connectivity (see
next step) before we add anything into the committed storage. We build the index
and unpack all deltas into this local directory, verifying each object's SHA as
it is indexed (we should see if we can leverage git's canonical implementation
for this). The temporary directory is deleted when the request completes
(success or failure) and the staging root is wiped on startup.

The next step will validate that all objects (and the objects reachable from
them) referenced in the commands `new-oid`s exist either in the received pack
or in SlateDB, rejecting the push if an object resolves against neither. If an
object points to an object that already exists in the SlateDB store, then we
do not need to walk that tree.

At this point, we can promote the objects to SlateDB (inlining small blobs and
offloading larger blobs to S3). Promotion writes children before parents
(blobs, then trees bottom-up, then commits), so committed storage only ever
contains objects whose full closure is already present — even if the server
dies mid-promotion. Anything promoted by a push that never reached its ref
update is dangling and left to garbage collection. Afterwards, we can update
the commit graph to make sure that once the commit ref is persisted reads can
properly query the new data.

Finally, the ref updates are CAS in a SlateDB transaction to make sure
that the ref's current value is `old-oid` for each command before updating
the reference's value to `new-oid`.
